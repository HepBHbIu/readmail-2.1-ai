# Readmail v2

Система автоматической обработки писем-претензий от клиентов (возвраты автозапчастей, оптовая торговля).
Читает входящую почту → понимает, что за обращение → извлекает структурированные данные → отдаёт чистый
JSON в 1С. Где может — разбирает бесплатно паттернами, где не хватает — дочитывает ИИ (текст и фото/сканы).

---

## Вход → выход (что на входе, что на выходе)

**Вход:** входящие письма по IMAP — текст, расклеенные HTML-таблицы, вложения (фото, PDF, Excel/акты),
внешние ссылки на фото/страницы возврата.

**Выход:** чистый документ возврата (`readmail-1c-v2`) для 1С:
```json
{
  "event": "return_ready",
  "buyer":   {"code": "...", "name": "..."},
  "claim":   {"kind": "wrong_item", "kind_label": "Пересорт", "number": null},
  "document":{"number": "82281", "date": "25.05.2026", "type": "УПД"},
  "items":   [{"part_number":"KR0399M","brand":"KRAUF","product_name":"Модуль…","quantity":1,
               "price":null,"received_part_number":null}],
  "comment": "...",
  "flags":   {"defect_documents": {...}, "has_photo": false, "has_document": false},
  "status":  {...}, "text": {"subject":"…","body":"…"}, "attachments": [...], "source": {...}
}
```
Каждый необязательный блок (comment/price/flags/status/text/attachments/source) включается тумблером —
несём всё нужное, лишнее выключаем.

## Как работает (конвейер)

```
IMAP-почта
  ① ИМПОРТ           imap_importer.py + email_parser.py
       raw_emails + вложения на диск (дедуп по message_id)
  ② ПАТТЕРНЫ (0 ток.) classifier.py — regex + пер-клиентские YAML-шаблоны
       → поля (артикул/бренд/имя/кол-во/документ) + тип претензии (claim_kind)
       → quality_gate.py: значения должны быть подтверждены контекстом (evidence-gate)
  ③ ИИ               ai_client.py — что паттерны не добрали, дочитывает модель
       режимы: «паттерн + ИИ-хвост» либо «полный ИИ» (каждое письмо)
  ④ БРАК (vision)    при claim_kind=defect — авто-прогон фото/сканов: ищет 3 документа брака
       (наряд установки, наряд снятия, акт сервиса) → defect_doc_flag
  ⑤ СВЕРКА           оператор проверяет/правит; в режиме обучения всё идёт через Сверку
  ⑥ ЭКСПОРТ          чистый payload v2 → outbox → доставка (JSON-файл / HTTP / локальный приёмник)
```

**Типы претензий:** брак, некондиция, пересорт, недовоз, перепоставка, некомплект, замена артикула,
корректировка, маркировка, отказ клиента. **Минимум для 1С:** документ+артикул+кол-во+тип (строго),
для отказа до поставки — номер обращения+артикул+кол-во.

Подробно про алгоритм, промты и паттерны: **[`docs/architecture_ai_patterns.md`](docs/architecture_ai_patterns.md)**.

## Два способа обработки: паттерны vs полный ИИ

Один и тот же результат (поля для 1С) можно получить двумя путями — разница в том, кто извлекает и
сколько это стоит.

| | **Паттерн-конвейер** (паттерны + ИИ-хвост) | **Автопилот ИИ** (full_ai) |
|---|---|---|
| Извлекает | regex/YAML по каждому клиенту; ИИ добивает хвост | модель читает **каждое** письмо |
| Промт | полный `SYSTEM_PROMPT` (спека + примеры) | компактный (схема + текст), вход −~54 % |
| Покрытие | ~85–90 % писем — **0 токенов**, детерминированно | 100 %, но каждое письмо платное |
| Плюс / минус | дёшево и проверяемо · надо вести YAML клиентов | не нужны паттерны · платим за всё |

**Прикидка стоимости** (на 1000 писем, deepseek-chat 9 ₽/млн вход · 3.72 ₽/млн выход):

| Режим | Через ИИ | ≈ Стоимость / 1000 писем |
|---|---|---|
| Паттерн-конвейер | ~15 % (хвост, полный промт) | **≈ 6 ₽** |
| Автопилот ИИ | 100 % (компактный промт) | **≈ 8–9 ₽** |

По деньгам близко, но паттерны дают 85 % бесплатно и без галлюцинаций — обычно основной режим, ИИ — хвост.
**Брак** (документы по фото) считается отдельно: vision-модель, ~0.15 ₽ за письмо с ~14 фото. Подробнее с
разбором структур и промтов — в [`docs/architecture_ai_patterns.md`](docs/architecture_ai_patterns.md).

### Схема работы

```
                          ┌──────────────┐
                          │  IMAP-почта  │  письма + вложения (фото / PDF / Excel / ссылки)
                          └──────┬───────┘
                                 │  импорт (дедуп по message_id)
                                 ▼
 ── Режим A · паттерны + ИИ-хвост ─────────────────────────────────────────────
                         ┌───────────────┐   0 токенов
                         │   ПАТТЕРНЫ     │   regex + YAML клиента
                         └───────┬───────┘
                                 ▼
                         ┌───────────────┐
                         │   ВАЛИДАЦИЯ    │   evidence-gate
                         └──┬─────────┬───┘
                подтверждено │         │ неполно
                            ▼         ▼
                        ┌──────┐  ┌─────────┐   SYSTEM_PROMPT
                        │  1С  │  │ ИИ-хвост│
                        └──────┘  └────┬────┘
                                       ▼  валидация
                                 ┌─────┴──────┐
                    подтверждено │            │ не классифицировано
                                 ▼            ▼
                          СВЕРКА → 1С    НЕРАЗОБРАННЫЕ (ручной разбор)

 ── Режим B · полный ИИ ───────────────────────────────────────────────────────
                         ┌───────────────┐   компактный промт
                         │  ИИ (каждое)  │
                         └───────┬───────┘
                                 ▼  валидация
                           ┌─────┴──────┐
                подтверждено│            │ не классифицировано
                           ▼            ▼
                       СВЕРКА → 1С   НЕРАЗОБРАННЫЕ (ручной разбор)

 ── Брак (claim_kind = брак / некондиция; в обоих режимах) ─────────────────────
      кейс брака ─► VISION по документам/фото (DEFECT_DOC_PROMPT)
          ├─ документы найдены (наряд установки/снятия, акт сервиса) ─► флаг документов в 1С
          └─ не найдены / файлов нет                                  ─► Сверка (ручной разбор)
```

Правило маршрутизации: подтверждено валидацией → **1С**; неоднозначно → **Сверка** (подтверждение
оператором); не классифицировано → **Неразобранные** (ручной разбор). Для брака флаг документов в 1С
ставится только при найденном комплекте подтверждающих документов.

## Паттерны (извлечение без токенов)

`app/classifier.py`: `FIELD_REGEX` (поля по меткам), `detect_kind` (тип претензии), `extract_fields`
(regex → фразовые шаблоны → YAML клиента → таблицы «| » → split бренд/артикул), мультипозиция, документы
брака. Пер-клиентские форматы — в `config/buyers/<code>.yml` (`templates`, `item_templates`, `ai_prompt`).
Цель — закрыть 85–90 % писем без ИИ. Кнопка «Обучить» дописывает выученные regex в YAML клиента.

## Валидация (Evidence Gate) — «понял» ≠ «можно в 1С»

`app/quality_gate.py` + модули доказательств проверяют, что каждое поле **подтверждено контекстом**
(иначе в 1С уйдёт мусор). И паттерны, и ИИ проходят гейт **одинаково** — он владеет `ready_for_export`:
- `part_number_evidence` — артикул по метке/строке товара/колонке (телефон/дата без метки → слабое);
- `document_number_evidence` — № реализации (УПД/накладная) ок; № претензии/заявки — блок;
- `quantity_evidence`, `claim_kind_evidence`, `buyer_evidence` — кол-во, причина, покупатель.
Не прошло → кейс в **Сверку** (оператор), не в 1С.

## ИИ: запрос и промты

**Запрос** (`app/ai_client.py`): `POST {base_url}/chat/completions` (OpenAI-совместимо, `Bearer`),
тело `{model, messages:[system,user], temperature:0, max_tokens, response_format:json_object}`.
Каждый вызов → живой лог (`/api/ai/live-log`) и учёт токенов (`ai_usage`, отчёт `/api/ai/token-report`).
Ответ ИИ не доверяется напрямую — снова проходит валидацию (выше).

**Промт извлечения** — `SYSTEM_PROMPT`: правила полей + примеры «письмо → JSON», напр.:
```
[«…отказа конечного покупателя RF5161S ROCK FORCE Набор ключей в количестве 1 по документу №79324 от 22.04.26»]
→ {"event_type":"new_return","claim_kind":"quality_refusal","fields":{
     "document_number":"79324","document_date":"22.04.2026","part_number":"RF5161S",
     "brand":"ROCK FORCE","product_name":"Набор ключей","quantity":"1","comment":"отказ конечного покупателя"}}
```
Правила промта: не брать заголовки таблиц/слова-метки как значения; brand НЕ из email/домена и не из марки
авто; № заявки ≠ № документа; дата документа ≠ дата письма; цитаты ниже (`>`, «От:») — только контекст.
К нему добавляется `ai_prompt` клиента. В режиме «полный ИИ» — **компактный** промт (схема + текст, вход −~54 %).

**Промт брака (vision)** — `DEFECT_DOC_PROMPT` (`app/main.py`): «определи тип документа → `{doc_type:
install_order|removal_order|service_act|part_photo|other, confidence, reason}`».

### Vision — тот же клиент, смена модели

Фото/сканы идут через **тот же RouterAI**, но `run_vision_extraction` на время вызова **подменяет модель**
на vision (`AI_VISION_MODEL`, напр. `qwen/qwen3-vl-32b-instruct`) и шлёт картинку как `image_url` (base64).
Поэтому один прогон брак-кейса в логе = **1 текст (deepseek) + N vision (qwen-vl)**. Включается `AI_VISION_ENABLED`.

### Сортировка и диалоги

`detect_event_type` отделяет `new_return` (первое письмо с данными) от продолжений
(`followup_reminder`/`followup_dialog`), отчётов (`supplier_report`), корректировок и т.д.
`_auto_link_followup` цепляет напоминания/ответы к родительскому возврату (по треду / № документа / №
возврата) и наследует поля — поэтому «5 напоминаний» не плодят пустые кейсы.

### Провайдеры
| Провайдер | Модель по умолчанию | Заметки |
|---|---|---|
| **RouterAI** | `deepseek/deepseek-chat` (текст), `qwen/qwen3-vl-32b-instruct` (vision) | дёшево, быстро, хороший русский |
| GigaChat | GigaChat-2-Lite | данные в РФ |
| Яндекс GPT | yandexgpt-lite | данные в РФ |
| Локальная (OpenAI-совм.) | Qwen3-VL | бесплатно, без интернета |

## Экспорт в 1С

- **Чистая схема v2** (`apply_one_c_payload_profile`): основа (клиент/документ/позиции) + блоки по тумблерам
  `ONE_C_INCLUDE_*` (price, comment, flags, status, text, attachments, source). Профиль `debug` отдаёт
  полный payload с evidence/gate — для аудита, не для боевой 1С.
- **Форматы доставки** (`ONE_C_EXPORT_MODE`): `file` (JSON-файлы) · `http` (POST на endpoint) ·
  `local_receiver` (локальный тест-приёмник, без внешней 1С) · `both` · `off`.
- Поля «что отправлять» настраиваются в UI: **Настройки → «Поля для 1С — что отправлять»**.

## Конфиг клиента (YAML, `config/buyers/<code>.yml`)

```yaml
buyer:
  code: my_client_ru
  name: Мой Клиент
  aliases:
    domains: ['@myclient.ru']
    subject_contains: [возврат, претензия]
statuses:
  defect: [брак, дефект]
  wrong_item: [пересорт, не тот артикул]
fields:
  regex:
    part_number: ['Артикул\s*[:№]?\s*([A-Z0-9][A-Z0-9._/-]{2,40})']
    document_number: ['накладн[ао]й\s*№\s*([0-9]{4,})']
templates:        # фразовые шаблоны под формат этого клиента
item_templates:   # повторяющиеся строки позиций (мультипозиция)
ai_prompt: |      # индивидуальная подсказка для ИИ (дописывается к SYSTEM_PROMPT)
```
Новому клиенту ИИ создаёт профиль после нескольких писем; кнопка «Обучить» дописывает выученные regex
в его YAML.

## Архивация и почта

| Период | Что хранится |
|---|---|
| 0–90 дней | полные данные (тело + вложения) |
| 90–365 дней | только JSON-метаданные |
| > 365 дней | удаляется |

Настройки `ARCHIVE_FULL_DAYS` / `ARCHIVE_META_DAYS`. Почта на сервере **не трогается** — учёт по
внутреннему реестру `message_id`. Вложения лежат в `data/attachments/<raw_email_id>/`.
